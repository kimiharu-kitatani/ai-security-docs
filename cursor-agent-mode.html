Cursor AIエージェント活用ガイド

対象：Cursorで複数のAIエージェント（Claude Code、Codexなど）を並行して使ってみたい方
形式：よくある疑問 → 現状の公式情報 → 使い方の整理

出典について：本文中の公式はCursor公式ドキュメント・公式changelog・公式ブログで確認できた事実、推定・二次情報はWeb記事や一般論で公式一次情報では未確認のもの、フォーラムはCursor公式フォーラムでのユーザー・スタッフの投稿ベースの情報です。

STEP 1｜なぜ「Cursorのエージェントモード」が注目されるのか

📋 よくある背景

複数の案件・プロジェクトを並行して抱えるようになると、Claude CodeやCodexなど複数のAIエージェントを別々のウィンドウで同時に起動して手動で行き来する運用になりがちです。「画面を切り替えるのが大変」「マルチタスクさせると自分の方が疲弊する」という負担が生まれ、これがCursorの「1画面で複数エージェントをまとめて見られる」UIへの関心につながるケースが多く見られます。

🔍 整理すると

  • 複数エージェントの並行運用が日常化すると、ウィンドウ管理そのものが仕事になってしまう
  • Cursorの「Agents Window（エージェントウィンドウ）」は、複数エージェントを1画面にまとめて見られるため、この負担を直接解消できる可能性がある
  • あわせて、処理速度・コストの良いモデル（Composer）への関心や、AIコーディングツールの業界動向へのキャッチアップ意識も動機になりやすい

📌 STEP 1 の結論

「複数のAIエージェントを並行運用したいが、ウィンドウ管理が大変」という悩みは、Cursorのエージェントモードに関心を持つ最も典型的なきっかけです。
まずはこの悩みが実際に解消されるのか、次のSTEPで公式情報をもとに整理します。

STEP 2｜New AgentモードでClaude Code / Codexを使うには

📋 よくある質問

「New Agentモードで普通に話すと、Cursor独自のモデル（Composerなど）が使われてしまう。Claude CodeやCodexといった他社製エージェントを使いたい場合はどうすればいいか？」

🔍 結論：整理が必要

やり方は大きく2つに分かれます。

方法1：New Agentのモデル切替

New Agent作成後、チャット欄のモデルのドロップダウンで、Cursor独自モデルのComposer以外に、Claude系（Sonnet/Opus）やGPT-5-Codex系などのモデルも選べます。公式

ただし注意点：これは「そのモデルを頭脳として使うCursor自身のエージェント」であって、Claude Code・Codexという製品本体が動くわけではありません。似ているようで別物です。

方法2：本物のClaude Code/Codexを1画面に並べる

導入済みの拡張機能のパネル（Claude Code拡張・Codex拡張など）を、Cursorのウィンドウ内にドラッグして並べる方法です。「Cursorのネイティブチャットを上段、Claude CodeとCodexのパネルを左右、下段にターミナル」という配置で1ウィンドウに集約している運用例が第三者の解説記事で具体的に確認できています。推定・二次情報

一度この配置を作れば、別ウィンドウで複数エージェントを手動で行き来する運用は不要になります。

🔍 公式情報で確認できたこと / できなかったこと

項目 状況
New Agentのモデル選択欄でClaude系・GPT-5-Codex系などを選べる 公式で確認済み（cursor.com/docs/models）
Agents Window（エージェントウィンドウ）で複数エージェントをローカル/クラウド/リモートSSH横断で並行実行できる 公式で確認済み（cursor.com/changelog/3-0）
New Agentタブの中で、拡張機能側のClaude Code/Codex本体をそのまま起動・表示できる 未確認。公式changelog・ドキュメントに該当の記載はなく、現時点ではできないと考えるのが妥当
拡張機能パネルをドラッグして1ウィンドウに集約する運用 推定・二次情報（第三者の解説記事で具体例を確認）

📌 STEP 2 の結論

Cursor自身のAgents Windowが管理できるのは、Cursor自身のエージェントのみです。
Claude Code・Codexという製品本体を1画面にまとめたい場合は、New Agentのモデル切替ではなく、拡張機能パネルの並列配置（方法2）が現実的な選択肢です。

STEP 3｜エージェントタブ・エージェントウィンドウの関係

📋 よくある疑問

「エージェントウィンドウ・サイドバーのチャット・エージェントタブ・New Agentボタンの関係がよく分からない」

これはCursor側のUIがここ数ヶ月で変わった影響もあり、ユーザーの間でも「よく分からない」という声が多い部分です。Cursor公式フォーラムでも同様の質問に対してスタッフの説明が限定的で、混乱・不具合報告が複数見られました。フォーラム UIがまだ発展途上であることが大きく、利用者の理解不足だけが原因ではないと考えられます。

🔍 大枠のイメージ

  • エージェントウィンドウ（サイドバー） = 起動中の全エージェントの一覧。ローカル/クラウド問わず一目で見える場所
  • エージェントタブ = そのウィンドウの中で、個々のエージェントの会話・作業状況を1つずつ表示するタブ。タブごとにモデルや作業対象（worktree）を変えられる
  • New Agent = 新しいタブ（新しいエージェントセッション）を1つ追加するボタン

「別ウィンドウで複数起動して手動で行き来する」運用から、「1つのエージェントウィンドウの中でタブを並べる」運用に置き換えるイメージで捉えると分かりやすくなります。

📌 STEP 3 の結論

Cursor 3.0（2026年4月2日リリース）で導入された「Agents Window」が、複数エージェント並行管理の核になっています。公式
Agent Tabsにより複数のエージェントチャットをサイドバイサイド/グリッドで表示可能で、各タブは個別にモード・モデル・worktreeを持てます。公式

STEP 4｜Composerモデルの使いどころと料金

🔍 Composerとは

Composerは速度とコストパフォーマンスに寄せたCursor独自のモデルです。モデル選択肢には、Cursor独自のComposer（1〜2.5）、Anthropic Claude系（Sonnet/Opus/Haiku各バージョン）、OpenAI GPT系（GPT-5系、GPT-5-Codex系）、Gemini、Grok、GLM、Kimiなどが並び、「Auto」で自動選択も可能です。公式（cursor.com/docs/models）

料金

モデル 入力 出力
Composer 2.5（標準） $0.50 / M トークン $2.50 / M トークン
Composer 2.5 Fast $3.00 / M トークン $15.00 / M トークン

公式（cursor.com/blog/composer-2-5）

🔍 使い分けの目安

  • 定型的な修正や量をこなす作業、スピード重視のタスク → Composerに任せる
  • 設計判断が必要な複雑なタスク、じっくり考えさせたい場面 → Claude系モデルを選ぶ

📌 STEP 4 の結論

Composer 2.5は競合の最上位モデルに比べて大幅に低コスト（標準版は競合フロンティアモデルの約1/10のコストとされる）で、コスト効率を重視するタスクに向いています。推定・二次情報（価格情報自体はCursor公式ブログに基づく）

STEP 5｜世の中でCursor / AIコーディングエージェントが使われる理由

🔍 一般的に挙げられる理由

  • 複数ファイルにまたがる編集・リファクタリングを自動でこなせる点が評価されている。Cursor公式ブログでは、Cursorのエージェントがデフォルト運用になった企業でPRのマージ数が39%増加したという自社発表がある。推定・一般論（提供元の自己申告データ）（出典：cursor.com/blog/productivity）
  • Stack Overflow 2025年調査では開発者の84%がAIツールを利用中/利用予定、51%のプロフェッショナル開発者が日常的にAIを利用していると報告されている。JetBrainsの2026年1月調査ではCursorの業務利用率は18%とされる。推定・一般論（二次情報、一次URLは未取得）
  • 「1画面で複数のエージェントを並行管理できる」UIは、Claude CodeやOpenAI Codexのようなターミナル中心のエージェントに対するCursor側の差別化ポイントとして、2026年4月のCursor 3.0で導入された。公式（cursor.com/changelog/3-0）

📌 STEP 5 の結論

Cursorのエージェント機能は「複数ファイル横断編集の自動化」と「複数エージェントの並行管理」の2点で差別化を図っている、というのが業界的な位置づけです。

注意点｜Cursorのエージェントモードは万能ではない

複数エージェントを並走させた後の成果物のすり合わせや、最終的な整合性チェックは人間側で必要になります。並行運用で楽になる部分と、逆に確認の手間が増える部分の両方があるという前提で導入を検討してください。

導入時のおすすめの進め方

1. まず1つの簡単なタスクで、エージェントタブを2〜3個並べて動かしてみる
2. 拡張機能パネル（Claude Code/Codex等）の並列配置を試し、自分に合う画面構成を見つける
3. 複数エージェントの成果物を統合する際のチェック手順を決めておく
4. 高リスクな操作（削除・送信・課金など）は必ず人間が最終確認する
