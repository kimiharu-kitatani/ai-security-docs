GitHub入門 — なぜ使うのか・どう使うのか

対象：複数人（そして複数のAIエージェント）で同じプロジェクトフォルダを触り始めたチーム
形式：解説 → 2人で行うハンズオン → 1人で行うハンズオン → よくある質問

参考教材：侍テラコヤ Git/GitHubカテゴリ（図解・ステップ形式で基礎からチーム開発まで学べる教材群。より体系的に学びたい方はこちらもどうぞ）

STEP 1｜なぜGitHubなのか

📋 よくある疑問

「OneDriveやGoogleドライブでフォルダを共有しているのに、なぜわざわざGitHubが要るの？」

🔍 クラウド同期フォルダとGitHubの違い

観点 クラウド同期（OneDrive / Googleドライブ） GitHub
反映のタイミング 保存した瞬間に全員のフォルダへ即反映 プッシュし、プルリクエストを承認して初めて共有版に反映
同時に編集したら 知らない間に上書き・「競合コピー」ファイルが発生 ブランチで作業が分離され、合流（マージ）時に内容を確認できる
変更の履歴 限定的（誰が何をなぜ変えたか追いにくい） 全変更が「誰が・いつ・何を・なぜ」付きで永久に残る
昔の状態に戻す 難しい・部分的 いつでも任意の時点へ丸ごと戻せる
AIエージェントとの相性 AIがファイルを変更すると即座に共有版が書き換わる（事故のもと） AIの変更もプルリクエストで人間が確認してから合流できる

一言で言うと：GitHubは「即時同期の共有フォルダ」ではなく「変更を確認してから合流させる仕組み」です。
複数の人＋複数のAIエージェントが同じプロジェクトを触る時代には、この「確認してから合流」が安全装置になります。

⚠️ 重要な注意：同期フォルダの中でGit管理をしない

❌ やってはいけない構成
OneDrive/
└── プロジェクトフォルダ/   ← ここをGit管理する
    └── .git/              ← OneDriveの同期処理とGitの管理情報が
                              ぶつかって壊れることがある

✅ 推奨の構成
C:\work\プロジェクトフォルダ\   ← 同期対象外のローカルに置いてGit管理
（共有・バックアップの役割はGitHub側が担うので、二重に同期する必要はない）

📌 STEP 1 の結論

「同時に触っても壊れない」「誰が何をしたか分かる」「いつでも戻せる」。この3つが揃うのがGitHubを使う理由です。
そしてGit管理するフォルダは、OneDrive等の同期フォルダの外に置きます。

STEP 2｜基本用語（この11個だけ覚えればOK）

用語 意味 例えるなら
リポジトリ プロジェクト一式の保管庫。GitHub上（リモート）と手元のPC（ローカル）の両方にある プロジェクトの「本棚」
クローン GitHub上のリポジトリを丸ごと自分のPCへ複製すること（最初の1回だけ） 本棚ごとコピーして持ち帰る
コミット 変更内容に説明を付けて記録すること ゲームの「セーブポイント」
プッシュ 手元のコミットをGitHubへ送ること アップロード
プル GitHub上の最新状態を手元に取り込むこと ダウンロードして手元を最新化
ブランチ 共有版に影響を与えずに作業できる、自分専用の作業レーン 本線から分かれた「支線」
プルリクエスト（PR） 「この変更を共有版に取り込んでください」という申請 承認をもらうための「稟議書」
マージ 申請された変更を共有版に合流させること 稟議の承認・実行
コンフリクト 2人が同じ箇所を別々に変えたときの衝突。どちらを採用するか人間が決めて解消する 書類の同じ欄に2人が別のことを書いた状態
.gitignore 「これはGitHubに上げない」と指定するファイル 持ち出し禁止リスト
main 共有版の本線ブランチ。ここが「チームの正」 マスターデータ

🔍 全体の流れ（図）

            ［GitHub：クラウド上の共有リポジトリ（main）］
                 ↑ プッシュ              ↓ クローン／プル
        ┌─────────────────┐    ┌─────────────────┐
        │ Aさんのパソコン │    │ Bさんのパソコン │
        │  ブランチで作業  │    │  ブランチで作業  │
        └─────────────────┘    └─────────────────┘
                 └── プルリクエスト → 相手が内容を確認 → マージ ──┘
                       （承認されて初めて main に反映される）

📌 STEP 2 の結論

日常の流れは「プル（最新化）→ ブランチで作業 → コミット → プッシュ → プルリクエスト → マージ」の繰り返しです。
この1周をSTEP 4・5のハンズオンで実際に体験します。

---

STEP 3｜GitHubに上げてはいけないもの

❌ 上げてはいけないもの
- .env ファイル・APIキー・パスワード類
- 顧客の個人情報を含む書類（氏名・住所・保険番号など）
- node_modules などの巨大な自動生成フォルダ
- 契約書・見積もりなどの機密書類（リポジトリの目的に不要なもの）

.gitignore の設定例

# .gitignore
.env
.env.*
node_modules/
secrets/
*.pem
*.key

プライベートリポジトリでも「共同作業者全員が見られる」前提で考えてください。
個人情報・シークレットの詳しい扱いは「📋 開発ガイド」「✅ リリース前チェック」タブを参照してください。

STEP 4｜ハンズオン①：2人で行う流れ確認（約60分）

📋 準備

  • 2人ともGitHubアカウントを持っていること
  • 役割を決める：【リーダー】＝リポジトリの持ち主／【メンバー】＝参加者
  • 練習用なので、実案件のフォルダではなく新しく作った練習フォルダで行うこと
  • AIエージェント（Claude Code等）に日本語で指示して進める前提で、各手順に「AIへの指示例」を付けています。もちろん自分でコマンドを打ってもOKです

🔨 手順

手順1【リーダー】プライベートリポジトリを作る

GitHubの画面右上「＋」→「New repository」→ 名前を入力（例：team-practice）→ Private を選択 →「Create repository」。

手順2【リーダー】練習用フォルダをプッシュする

PCに練習用フォルダを作り、中にテキストファイルを1〜2個置いてから、AIエージェントに指示します。

AIへの指示例：
「このフォルダをGitHubのリポジトリ team-practice にプッシュしてください。
 その前に .gitignore を作って node_modules と .env を除外してください。
 実行する前に、何をするか説明してください」

ポイント：「実行する前に説明して」を付けると、AIが何をしようとしているか確認してから進められます。

手順3【リーダー】メンバーを招待する

GitHubのリポジトリ画面 →「Settings」→「Collaborators」→「Add people」→ メンバーのアカウント名を入力して招待。メンバーは届いた招待メールを承認します。

手順4【メンバー】クローンする

リポジトリ画面の緑の「Code」ボタンからURLをコピーして、AIに渡します。

AIへの指示例：
「このURLのリポジトリを C:\work\team-practice にクローンしてください。
 （URLを貼り付け）」

※ 保存先はOneDrive等の同期フォルダの外にしてください（STEP 1参照）。

手順5【メンバー】ブランチを作って編集する

AIへの指示例：
「新しいブランチを作って、README.md に自己紹介を1行追加して、
 コミットしてプッシュしてください」

手順6【メンバー】プルリクエストを作る

AIへの指示例：
「今の変更でプルリクエストを作ってください。
 変更内容の説明も書いてください」

手順7【リーダー】プルリクエストを確認してマージする

GitHubのリポジトリ画面 →「Pull requests」タブ → 届いたプルリクを開く →「Files changed」で何がどう変わるのかを目で確認 → 問題なければ「Merge pull request」。気になる点があればコメントを書いて修正を依頼します。

手順8【両者】最新を取り込む

AIへの指示例：
「mainの最新を取り込んでください」

これでメンバーの変更が、リーダーの確認を経て、2人の手元に揃ったことになります。この1周がGitHubの基本サイクルです。

手順9（応用）わざとコンフリクトを起こして解消する

2人が同じファイルの同じ行をそれぞれのブランチで別々に変更し、順番にプルリクを出してマージすると、2人目のマージで衝突（コンフリクト）が起きます。

AIへの指示例：
「コンフリクトが起きています。両方の変更内容を見せてください。
 その上で、○○の方を採用して解消してください」

コンフリクトは故障ではなく、上書き事故を防いでくれた安全装置です。慌てずに「どちらを採用するか」を人間が決めれば解消できます。

✅ 2人版チェックリスト（お互いに声に出して確認）

  • [ ]【メンバー】プッシュしただけでは共有版（main）が変わらないことを確認した
  • [ ]【リーダー】相手の変更がプルリクで届き、自分が承認するまでmainに入らないことを確認した
  • [ ]【両者】マージ後にプルすると、2人の手元が同じ状態になることを確認した
  • [ ]【両者】GitHubの「Commits」画面で「誰が・いつ・何を」変えたか履歴を見た
  • [ ]【両者】コンフリクトを体験し、慌てず解消できた
  • [ ]【両者】.gitignoreで除外したファイルがGitHub上に無いことを確認した

STEP 5｜ハンズオン②：1人で行う練習（約30分）

1人で「リーダー役」と「メンバー役」の一人二役をやる練習です。相手がいなくても流れを一通り体験できます。

🔨 手順

  1. リポジトリ作成：GitHubで新しいプライベートリポジトリ（例：solo-practice）を作る
  2. クローン：AIに「このURLをローカルの◯◯フォルダにクローンして」と指示
  3. 編集→コミット→プッシュ：ファイルを1つ作って「変更をコミットしてプッシュして」と指示
  4. 履歴を見る：GitHubの「Commits」画面で、今の変更が記録されていることを確認
  5. ブランチで変更：「新しいブランチを作って◯◯を変更し、プッシュして」と指示
  6. プルリク→セルフマージ：「プルリクエストを作って」→ GitHub画面で自分で内容確認 → 自分でマージ（1人でもこの確認の一呼吸が練習になります）
  7. プル：「mainの最新を取り込んで」と指示して手元を最新化
  8. 戻す体験：「さっきのコミットを取り消して、1つ前の状態に戻して」と指示（revert体験）。「いつでも戻せる」を一度体験しておくと、以後怖くなくなります

✅ 1人版チェックリスト

  • [ ] クローン→編集→コミット→プッシュの流れを自分で(AI経由で)一周できた
  • [ ] GitHubの画面でコミット履歴と差分（何が変わったか）を見られた
  • [ ] ブランチ→プルリク→マージの流れを体験した
  • [ ] コミットの取り消し（revert）で過去の状態に戻せた
  • [ ] 「プッシュ＝即共有ではない」「マージして初めて共有版が変わる」を自分の言葉で説明できる

STEP 6｜よくある質問

Q. 間違って作ったリポジトリを消したい

リポジトリ画面 →「Settings」→ 一番下の「Danger Zone」→「Delete this repository」→ 確認のためリポジトリ名を入力して削除します。「とても大変なことだよ」と何度も確認されるのは、それだけ取り返しがつかない操作だからです。

Q. プライベートリポジトリなら個人情報を上げてもいい？

おすすめしません。共同作業者全員が見られますし、誤って公開設定にする・アカウントが侵害されるなどの事故リスクが残ります。個人情報・シークレットはそもそも上げない設計にしてください（STEP 3参照）。

Q. AIエージェントにGit操作を任せるときの注意は？

プッシュ・マージ・削除・強制上書き（force push）など「共有に影響する操作」は、実行前に何をするか説明させてから承認する習慣をつけてください。「実行する前に説明して」の一言をプロンプトに足すだけで効果があります。

Q. NotionやGoogleドライブとはどう使い分ける？

ツール 向いているもの
GitHub コード・設定ファイル・AIへの指示書など「変更履歴と承認フローが欲しいもの」
Notion等のDB 顧客データベース・タスク管理・日常のメモ
Googleドライブ等 大きなファイルの受け渡し・共有（動画・PDF資料など）

全部をGitHubに寄せる必要はありません。「同時に編集して壊れると困るテキスト系のもの」から順にGitHubへ移すのが現実的です。

📌 まとめ

GitHubは「変更を確認してから合流させる仕組み」。
複数の人と複数のAIが同じプロジェクトを触る時代の、上書き事故を防ぐ安全装置です。
まずはSTEP 5の1人練習で一周し、その後STEP 4の2人ハンズオンをチームでやってみてください。
