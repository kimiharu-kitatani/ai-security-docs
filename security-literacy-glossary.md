# セキュリティ・リテラシー用語集
## クライアントとの会話で押さえる言葉と論点

**目的**：細かい技術名を暗記することではなく、リスクを整理し、優先順位を決め、相手が安心して相談できる状態を作ること。

> **会議での基本姿勢：**  
> 「何が危険か」だけでなく、「どの業務で、どの情報が、どこを通り、誰に見えるか」を一緒に確認する。

---

## 最初に聞く5つの質問

```
1. この業務で扱う一番守るべき情報は何ですか？
2. その情報はどこに保存され、誰がアクセスできますか？
3. 外部サービスやAIに送ってよい情報・送ってはいけない情報は分かれていますか？
4. 退職者・外注先・共有アカウントのアクセスは管理できていますか？
5. 事故が起きた時、誰が気づき、誰が止め、誰に報告しますか？
```

---

## 押さえておくキーワード

| キーワード | 一言でいうと | 会議での使い方 |
|---|---|---|
| 最小権限 | 必要な人に、必要な範囲だけ許可する | 「全員管理者」になっていないか確認する |
| ゼロトラスト | 社内だから安全、と決めつけない | 端末・ユーザー・場所ごとに確認する考え方として説明する |
| MFA / 2FA | パスワード以外の確認を足す | 重要サービスは認証アプリ以上を推奨する |
| パスキー | パスワードを使わず端末認証でログインする仕組み | フィッシング対策として話題に出せる |
| SSO | GoogleやMicrosoftなど1つの認証基盤でログインを管理する | 退職時のアクセス停止をまとめやすい |
| RBAC | 役割ごとに権限を分ける | 管理者・一般担当・閲覧のみを分ける話に使う |
| 監査ログ | 誰が、いつ、何をしたかの記録 | 事故後に追えるか、普段から確認する |
| DLP | 機密情報の持ち出しを検知・防止する仕組み | 個人情報や顧客名簿の外部送信対策で使う |
| シャドーIT | 会社が把握していない外部ツール利用 | 便利ツール利用を責めず、棚卸しする入口にする |
| サプライチェーンリスク | 自社でなく取引先・利用ツール経由で起きるリスク | 外注先、SaaS、プラグイン、npm/pip依存の確認に使う |
| シークレット管理 | APIキー・パスワードを安全に保管すること | .env直書きから1PasswordやSecrets Managerへの移行を提案する |
| インシデントレスポンス | 事故発生時の初動手順 | 「誰に連絡するか」「何を止めるか」を事前に決める |

## 攻撃・脅威の用語

| 用語 | 意味 | まず見る対策 |
|---|---|---|
| フィッシング | 偽メール・偽サイトで認証情報を盗む | MFA、パスキー、URL確認、訓練 |
| スミッシング | SMSを使ったフィッシング | SMSリンクを開かない、公式アプリから確認 |
| ビジネスメール詐欺（BEC） | 取引先や経営者になりすまして送金させる | 振込先変更は電話など別経路で確認 |
| ランサムウェア | データを暗号化し、復旧と引き換えに金銭要求する | バックアップ、権限分離、EDR、初動手順 |
| マルウェア | 悪意あるソフト全般 | 更新、EDR、未知USB禁止、管理者権限制限 |
| 認証情報漏洩 | ID・パスワード・APIキーが外部に出る | ローテーション、MFA、パスワード使い回し禁止 |
| Evil Twin | 本物と同名の偽Wi-Fi | SSID確認、自動接続OFF、VPN、テザリング |
| BadUSB | USB機器がキーボード等に化けて操作する攻撃 | 不明USB禁止、離席時ロック、USB制御 |
| プロンプトインジェクション | AIに隠れた命令を読ませて意図しない動作をさせる | 外部入力を信用しない、ツール実行前確認、権限分離 |
| データポイズニング | 学習・参照データに悪い情報を混ぜる | 情報源管理、レビュー、RAGの参照元確認 |

## AI導入時に強い人に見える論点

```
□ AIに送ってよい情報・送ってはいけない情報を分類する
□ APIキーや個人情報がプロンプト・ログ・スクショに混ざらない運用にする
□ AIエージェントに読ませないファイルを明確にする
□ ツール実行・外部送信・ファイル削除は人間の確認を挟む
□ 生成物は「正しさ」「権限」「公開範囲」の3点でレビューする
□ 社内ルールは厳しくしすぎず、現場が守れる形に落とす
```

---

## 会議で使える言い回し

```
「まず脅威を増やすより、守るべき情報と入口を整理しましょう」
「このリスクは技術対策より、運用ルールでかなり下げられます」
「VPNは通信経路の対策で、AIに送った内容そのものは守れません」
「便利さを落としすぎると守られないので、現場の導線に合わせます」
「事故が起きない前提ではなく、起きた時に止められる形にしましょう」
「共有アカウントは、誰が操作したか追えないのが一番の問題です」
```

## クライアントに見せる優先順位

| 優先度 | 見るポイント | 理由 |
|---|---|---|
| 1 | MFA、退職者アカウント、共有アカウント | 乗っ取りと内部不正の入口になりやすい |
| 2 | APIキー、.env、パスワード共有 | 漏洩すると被害が即時に広がる |
| 3 | 個人情報・顧客情報の保存場所 | 法務・信用への影響が大きい |
| 4 | バックアップと復旧手順 | ランサムウェアや誤削除から戻れるかが決まる |
| 5 | ログと通知 | 事故に気づけない状態を避ける |

> **一言まとめ：** 用語をたくさん知っていることより、「守る情報」「入口」「権限」「ログ」「初動」を順番に確認できることが、セキュリティに強い人の見え方を作ります。
